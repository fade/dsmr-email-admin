* Scope

Day-to-day operation of a DSMR qmail/vpopmail/dovecot mail host: keeping the
stack healthy, authenticating outbound mail so it reaches Gmail/Yahoo, and
keeping the sending IP's reputation clean. The =dsmr-email-admin= tool is the
single entry point for all of this; this manual is the why and the when behind
its subcommands.

Run everything as root on the mail host. Every mutating action backs up the
file it changes (=<file>.bak-<timestamp>=) and acts only when its detector
fires, so a value set deliberately is left alone.

* Architecture

This section is the map: the packages that make up the stack, the path a message
takes through it, every point where you can hook in or change behaviour, and the
control files that drive each stage. The operational sections (1–6) are the
procedures; this is the model they assume.

** Email flow

Two entry points (port 25 for the world, port 587 for authenticated submission)
converge on =qmail-smtpd=, which gates the message and hands it to =qmail-queue=.
=qmail-send= then routes each recipient to local, virtual (vpopmail), or remote
delivery. Retrieval is entirely separate: dovecot reads the same Maildirs.

** Hook points

Each is a deliberate place to change behaviour. They fire in the order a message
meets them.

1. *tcprules* (=control/tcp.smtp.cdb=, =control/tcp.submission.cdb=, compiled
   from the =tcp.smtp= / =tcp.submission= text sources). Per-source-IP policy:
   set =RELAYCLIENT= to let an IP relay without auth, deny abusive ranges, set
   per-IP env. On :25 only listed/trusted IPs get =RELAYCLIENT=; on :587 relay
   comes from AUTH, not the IP.
2. *Pre-greeting defences* — =SMTPD_GREETDELAY= (5s on :25, 3s on :587) and
   =DROP_PRE_GREET=1= hang up on bots that talk before the banner.
3. *HELO checks* — =HELO_DNS_CHECK=GNLR=.
4. *STARTTLS* — advertised whenever =control/servercert.pem= is present (this is
   STARTTLS in qmail-smtpd, not an =sslserver= wrapper; the run scripts use plain
   =tcpserver=). =control/servercert.pem= is shared with dovecot.
5. *SMTP AUTH* (:587) — =SMTPAUTH="!"= forces auth before relay; =AUTH_PROG= is
   =checkpassword-dovecot= when installed (one passdb covering vpopmail users,
   curated =/etc/dovecot/system-passwd= entries, and PAM), else =vchkpw=
   (vpopmail only). =FORCEAUTHMAILFROM=1= ties the envelope sender to the
   authenticated user.
6. *SPF* — =control/spfbehavior= (0 off … higher = stricter, up to reject).
7. *qmail-spp plugins* — only active when *both* =control/smtpplugins= and the
   =plugins/= directory exist (=ENABLE_SPP=). Per-stage plugin hooks
   (connection/helo/mail/rcpt/data). Leaving =ENABLE_SPP= unset is deliberate:
   any value triggers SPP init and fails closed if =smtpplugins= is absent.
8. *The RCPT gate (RCPTCHECK)* — the stack's main recipient policy. With
   =control/localfirst=1= it is =localfirst-rcptcheck= (system-first hybrid
   routing, see §4); otherwise chkuser plus the standalone
   =rcptcheck-overlimit= throttle. =RCPTCHECKRELAYCLIENT=1= so authenticated
   submission is still validated. This is where unknown recipients are rejected
   at RCPT instead of accepted-and-bounced (see §3).
9. *DATA size* — =control/databytes=.
10. *Routing* (=qmail-send=) — =control/locals= (system delivery),
    =control/virtualdomains= (vpopmail), =control/localfirstdomains= (hybrid),
    =control/rcpthosts= / =morercpthosts.cdb= (what we accept at all).
11. *System aliasing* — =~alias/.qmail-default = | fastforward /etc/aliases.cdb=.
    The cdb is the live alias database; rebuild it with =newaliases= after
    editing =/etc/aliases= (=newaliases= delegates to fastforward's
    =qmail-newaliases=). With localfirst enabled, an alias missing from the cdb
    is rejected at RCPT — keep the two in sync.
12. *Per-user / per-domain .qmail* — =~user/.qmail[-ext]=, and a vpopmail
    domain's =.qmail-default= (=| vdelivermail= or, on a hybrid domain,
    =| localfirst-dispatch HOST= which re-injects a system identity to the
    locals host and otherwise falls through to vdelivermail).
13. *Outbound filter* — =control/filterargs= turns on =spawn-filter= as
    =QMAILREMOTE=; that is where =qmail-dkim= signs outbound mail. Per-domain
    filter lines select the signer and key.
14. *SRS* — =control/srs_domain= (+ =srs_secrets=, =srs_maxage=, …) enables
    Sender Rewriting on forwarded mail so it survives the forwarding domain's
    SPF; absent =srs_domain=, SRS is off.
15. *Outbound transport* — =control/smtproutes= (smarthost / per-domain routes),
    =control/tlshosts= / =notlshosts= (force/skip outbound TLS),
    =control/helohost=, =control/outgoingip=.
16. *Retrieval (dovecot)* — =ssl = required=; auth and userdb against the
    vpopmail SQL backend plus =/etc/dovecot/system-passwd=; reads the vpopmail
    Maildirs directly.

* 1. Health and self-healing configuration

Several stack control files are written by a package postinst *only if absent*
— correct for files holding live credentials, but it means a host first
provisioned before a packaging fix keeps the old, broken value forever. The
recurring faults and their repair:

| Symptom                                             | Cause                                    | Fix              |
|-----------------------------------------------------+------------------------------------------+------------------|
| smtpd dies "failed to map segment" (libgssapi/krb5) | control/softlimit too low for OpenSSL3   | =fix softlimit=    |
| vpopmail/dovecot can't auth to PostgreSQL           | vpopmail.pgsql password/database swapped | =fix pgsql-config= |
| dovecot auth returns temp_fail                      | dovecot SQL conf out of sync             | =fix dovecot-sql=  |
| no virtual-user auth/delivery; lost after reboot    | vpopmail's pg cluster stopped/disabled   | =fix pg-cluster=   |

#+begin_src sh
dsmr-email-admin check                 # read-only audit of the whole stack
dsmr-email-admin fix --dry-run         # preview repairs
dsmr-email-admin fix                   # apply all detected repairs
dsmr-email-admin selftest              # prove delivery + dovecot auth end to end
#+end_src

*Run =check= after every package upgrade.* The postinsts do not self-heal an
existing file, so an upgrade can ship a fix that a long-lived host never adopts.

* 2. Sender authentication for virtual domains (SPF / DKIM / DMARC)

Since 2024 Gmail and Yahoo reject mail whose domain authenticates with neither
SPF nor DKIM (=550-5.7.26 ... the sender is unauthenticated=). On a multi-domain
host typically only the primary domain is signed, so every other hosted domain
is undeliverable to the big providers until it gets an SPF record and a DKIM key.

** What the tool does, and does not, do

It provisions the *signing* side (generates and installs the DKIM key so
qmail-dkim signs the domain's outbound mail immediately) and writes
copy-pasteable DNS records to =control/authdns/<domain>=. It *never writes to a
nameserver* — hosted domains use a mix of in-house and external-registrar DNS,
so the operator publishes the records by hand.

#+begin_src sh
dsmr-email-admin authdns status        # per-domain coverage: key / DKIM-DNS / SPF / DMARC
dsmr-email-admin authdns report        # narrative: who is covered, who is blocked, and why
dsmr-email-admin authdns provision <domain>     # install key + write the record file
cat /var/lib/qmail/control/authdns/<domain>     # paste these at the domain's DNS host
dsmr-email-admin authdns verify <domain>        # confirm published DNS matches the local key
#+end_src

** Care point: only provision domains that actually send through this host

A domain hosted here for *mail receipt* may *send* through an external service.
Its existing SPF will point elsewhere (or be a =-all= lockdown). Installing a
local key and publishing our records for such a domain *breaks its real mail*.
=authdns provision-all= therefore skips any domain that already publishes SPF or
DKIM (=--include-external= overrides). Before provisioning, sanity-check with
=authdns status=: a domain showing existing SPF/DKIM is managed elsewhere — leave
it alone.

Defaults: RSA-2048 keys, selector =default=, SPF =~all= (softfail, like the
primary domain). Use =spf suggest --hard= for =-all= once you are sure a domain
sends only through this host.

* 3. Outbound reputation and backscatter

The single biggest reputation killer on a multi-domain MTA is *backscatter*: the
host accepts mail to a local recipient that then *fails delivery*, and qmail
bounces it to the (usually forged) envelope sender. Spammers forge senders at
throwaway domains, so the host sprays misdirected bounces at those domains and
receivers flag the IP for "unsolicited mail". Backscatter is almost always a
*recipient-validation* problem: mail that should be rejected at RCPT is instead
accepted and bounced.

** Diagnose

#+begin_src sh
dsmr-email-admin reputation audit
#+end_src

Reports the share of the queue that is null-sender bounces (the backscatter
fingerprint), lists the accept-then-bounce recipients found in queued failure
notices, and classifies each as:

- *a real user with broken delivery* — the account exists but has no =Maildir=
  (e.g. an off-system forward was removed and nothing replaced it). Fix: create
  the user's Maildir so mail lands instead of bouncing
  (=maildirmake ~user/Maildir=, owned by the user, mode 700), or, if the account
  is defunct, stop its mail from bouncing.
- *a non-existent account* — accepted at RCPT and bounced at delivery. This is a
  gate gap; see the alias catch-all below.

The audit also flags a =~alias/.qmail-default= that runs =fastforward= without
=-p= (which bounces unknown recipients), and counts mailbox forwards to external
freemail (forwarding spam onward re-emits it from this IP and hurts reputation —
filter before forwarding, or do not forward spam).

** Test the recipient gate as a real sender

#+begin_src sh
dsmr-email-admin reputation probe nosuchuser@yourdomain.example
#+end_src

This connects to the host's *own public IP* (not 127.0.0.1, which is a
RELAYCLIENT and would accept anything), so it shows what an outside sender
actually gets. A =250= for a deliberately-invalid address means the gate
accepts-then-bounces — a backscatter faucet. A =550= means it rejects cleanly.
(Needed because a commercial ISP and many clouds block outbound port 25, so you
often cannot probe :25 from elsewhere; the loopback-to-public-IP path is the
reliable way to see the external view from the host itself.)

** Stop the bleeding, in order

1. *Close the faucet first.* Fix the broken mailbox, or stop the catch-all from
   bouncing — otherwise the queue refills as fast as you drain it.
2. *Then purge the backscatter:*
   #+begin_src sh
   dsmr-email-admin reputation purge            # dry-run: count
   dsmr-email-admin reputation purge --commit   # delete (briefly stops qmail-send)
   #+end_src
   Only null-sender =failure notice= messages are removed; real queued mail
   (which has a sender) is untouched.

** The fastforward alias catch-all (a classic system-domain faucet)

On a =locals= (system) domain, qmail routes any local-part that is not a real
=/home= user to the =alias= user. If =~alias/.qmail-default= is
=| fastforward <cdb>=, fastforward *bounces* any address not found in the cdb
(its default =-P= behaviour). So spam to =madeup@systemdomain= used to be
accepted (the catch-all exists) and then bounced — backscatter — by any RCPT
gate that treated "a catch-all exists" as "deliverable".

*This is now handled by localfirst (≥ 1.0-6).* When =~alias/.qmail-default= is a
=| fastforward <cdb>= line, =localfirst-rcptcheck= reads =<cdb>= directly and
reproduces fastforward's own verdict, so an unknown system address gets an
immediate *550 at RCPT* and is never accepted-then-bounced. It probes exactly the
targets fastforward does — for =lp@domain= the keys =:lp@domain=, =:@domain=,
=:lp@= (lowercased; the any-host alias form carries the trailing =@=). A =-p=
pass-through fastforward, or a cdb it cannot read, falls back to accept.

Two consequences for operators:

- *Keep the cdb in sync with =/etc/aliases=.* The cdb is the live alias database;
  after editing =/etc/aliases= run =newaliases= (it delegates to fastforward's
  =qmail-newaliases= to rebuild the world-readable =/etc/aliases.cdb=). A real
  alias present only in the text file but missing from the cdb is now *rejected*.
- *Do not run the old =-p= interim alongside this.* Adding =-p= turns the
  catch-all into pass-through, which localfirst treats as "accept everything"
  (no 550, mail silently dropped). Use the plain =| fastforward /etc/aliases.cdb=
  so localfirst's RCPT reject is in force.

* 4. The recipient-validation gate (localfirst)

=localfirst-rcptcheck= replaces chkuser as the sole RCPT gate, so it can accept
system-user and qmail-extension recipients that the vpopmail-only chkuser could
not see. It is *opt-in*: the smtpd/submission run scripts switch to it only when
=control/localfirst= first line is =1=. Installing the binary never changes
validation on its own — a host opts into system-first routing deliberately.

Per recipient: a domain not in rcpthosts is a relay target (accept; AUTH+tcprules
already enforced it); a =locals= domain validates against the system layer only
(real /home users, qmail extension addressing, ~alias/.qmail); a
=localfirstdomains= domain is hybrid (system first, then vpopmail); any other
local domain is pure vpopmail (=vrcptcheck=). A confirmed backend outage *defers*
(451), never rejects.

Known behaviours to keep in mind:

- *fail-open on private homes.* The validator runs as the smtpd user (vpopmail),
  which cannot traverse a mode-700 home, so it accepts every *extension* of such
  a user and lets delivery decide. On a host with world-traversable homes
  (751/755) extensions are validated precisely. Keep system homes 751, not 700,
  if you want precise extension rejection.
- *fastforward catch-all is validated against the cdb.* When
  =~alias/.qmail-default= is =| fastforward <cdb>=, the gate reproduces
  fastforward's own verdict: it reads =<cdb>= directly and accepts only
  recipients fastforward would forward (the targets =:lp@domain=, =:@domain=,
  =:lp@=), rejecting unknowns at RCPT (550) rather than accepting them and
  bouncing later. A =-p= pass-through fastforward, or an unreadable cdb, falls
  back to accept. Consequence: the cdb must track =/etc/aliases= — rebuild it
  with =newaliases= after editing aliases, or a newly-added alias is rejected
  (see §3 and the Architecture hook points).

* 5. Care-and-feeding checklist

- *After any package upgrade:* =dsmr-email-admin check= (postinsts do not
  self-heal existing config).
- *Adding a virtual domain that sends through this host:* =authdns provision
  <domain>=, publish the records, =authdns verify <domain>=.
- *Periodically, and after any "why is my mail in spam" report:* =authdns report=
  (authentication coverage) and =reputation audit= (backscatter).
- *Watch the queue:* =qmail-qstat=. A queue climbing into the hundreds/thousands
  of mostly-undeliverable messages is backscatter or a compromised account until
  proven otherwise — run =reputation audit=.
- *Never accept-then-bounce.* Reject unknown recipients at RCPT. A bouncing
  catch-all, a removed forward with no replacement mailbox, or an over-broad
  alias all turn into outbound backscatter that the receiving world blames on
  this IP.

* 6. Topology notes

- The mail host is the signing and sending MTA; DKIM keys live in
  =control/domainkeys/<domain>:<selector>= and outbound is signed by qmail-dkim
  via =control/filterargs= (=DKIMSIGNKEYDIR=).
- The vpopmail backend is PostgreSQL (=vpopmail.pgsql=); dovecot reads the same
  credentials via a generated SQL conf. Both are create-if-absent — see §1.
- The package repository is managed with reprepro on the build host and published
  to the production host by rsync; the production host is never built on.
